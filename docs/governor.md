# Governor design

## 2. 概要

**目的**
- DAOのガバナンス投票を備えたDAppsの詳細設計を示す
- DAO活動の透明性・信頼性向上のため、投票・提案等の行動履歴をブロックチェーン上で永続化する
- ウォレットアドレス単位でユーザーの行動履歴を収集・可視化を行う

**対象範囲**
- ガバナンス投票スマートコントラクト (OpenZeppelin Governance)
- 投票用フロントエンド（React＋Vite）
- 投票後NFT配布機構（投票結果によるNFT発行・転送）

## 3. システム全体構成
### 3.1 システム概要
- フロントエンド DApps（React＋Vite）
- ガバナンススマートコントラクト（OpenZeppelin Governor）
- NFT発行コントラクト（ERC-721）
- ガバナンストークンコントラクト（ERC-20 Votes）
- チャットメッセージ管理（IPFS）

### 3.2 開発環境
- フロントエンド：React, Vite, TypeScript, ethers.js
- テスト：Jest
- スマートコントラクト：Solidity, Hardhat
- ライブラリ：OpenZeppelin Contracts, Governor
- ストレージ：IPFS
- ウォレット連携：ethers.js Wallet

## 4. 機能設計

### 4.1 機能一覧（拡張部分）

| F-01 | ガバナンストークン発行（QR） | QRコードを通じてガバナンストークン（FT）を発行。ナンス値による一意性管理と署名検証を実施。 |

#### 4.2.1 F-01 ガバナンストークン発行
特定の場所やイベントで、固定のQRコードを通じてガバナンストークン（FT）を発行する。

- 入力: QRコードに埋め込まれたペイロード
- 出力: ガバナンストークン（FT）の残高増加

**前提条件**
- ユーザーはウォレットを接続済み

**処理フロー（概要）**
- フロントエンドで固定QRコードを読み取り、ナンス値を取得
- ペイロード（ナンス値・署名等）を検証
- コントラクト側でナンス値ごとに発行済みか管理し、未発行の場合のみFTを発行
- 成功時、トークン残高を更新

**ナンス値管理仕様**
- コントラクトは各ナンス値（nonce）ごとに発行済みかどうかを記録する（mapping(uint256 => bool) usedNonces など）
- 既に使用済みのナンス値でmintを呼び出した場合は拒否する

#### 4.2.2 F-02 提案作成
ガバナンストークン（FT）保有者が OpenZeppelin Governor コントラクトを利用して DAO に対する提案を作成できるようにする。

- 入力: 提案タイトル、説明、実行対象コントラクトアドレスなど
- 出力: 新規 Proposal IDのイベントログをスマートコントラクトで発行

**処理フロー（概要）**
1. フロントエンドで提案内容（タイトル・説明・実行先アドレス・call data等）を入力
2. 説明文など長文はIPFSに保存し、そのハッシュ（CID）を取得
3. Governorコントラクトの propose 関数（OpenZeppelin標準: propose(address[] targets, uint256[] values, bytes[] calldatas, string description)）を呼び出す
	- descriptionにはIPFSのCIDやタイトル等を含める
4. コントラクトからProposal IDが発行される
5. Proposal IDをUIに表示

**備考**
- OpenZeppelin Governorの仕様に準拠
- 提案作成には十分なガバナンストークン残高が必要
- [提案内容データ](#62-オフチェーン--ipfsデータ)はIPFSに記録する

#### 4.2.3 F-03 投票
トークン保有量に応じた投票を実現する。

- 入力: Proposal ID, 投票選択肢（賛成/反対/棄権）

**処理フロー**
1. 提案一覧から対象 Proposal を選択
2. 投票内容を選択し、castVote or castVoteWithReason を呼び出し
3. 投票結果は Governor の状態として集計d

## 5 UI設計

### 5.1 FT発行管理画面
- FT発行履歴管理
- 過去のFT発行履歴を表示
- ユーザーのウォレットアドレス
- 発行場所

### 5.2 提案・投票画面
- 提案一覧の表示
- 提案作成画面
- 提案内容の入力
- 提案

### 5.2 ユーザー行動履歴管理画面
- グラフデータベースと連携し、ユーザー行動履歴を可視化（ノード：ユーザー、エッジ：アクション）
- ユーザーごとの行動履歴タイムライン表示（投票・提案・チャット・NFT取得等）
- 貢献度・アクティビティグラフ（棒グラフ・円グラフ・ヒートマップ等）
- 行動履歴の検索・フィルタ（期間・アクション種別・ウォレットアドレス等）
- ユーザー詳細画面で、IPFS上の補助データや匿名化データも参照可能
- 管理者による表示制御・匿名化設定

## 6. データ設計（概要）
### 6.1 オンチェーンデータ
**FTコントラクト**
- ユーザー情報(mapping(userAddress -> uint[]))

**Governor**
- 提案情報
    - Proposal ID(uint)
    - 状態
    - 投票数(uint)
- 投票作成イベント
    - proposal ID
    - 提案名
- 提案の内部状態
    - 0: Pending(投票作成後は保留中で投票不可な状態)
    - 1: Active(投票が可能な状態)
    - 2: Cancel(投票がキャンセルされ終了した状態)
    - 3: Defeated(賛成少数で投票が終了した状態)
    - 4: Successed(賛成多数で投票が終了した状態)

### 6.2 IPFSデータ
- 提案詳細説明（JSON）
    - コントラクトアドレス: target(String)
    - 提案内容: text(String)
    - 提案後の実行内容: data(String)
    - 提案詳細のハッシュ値: descrptionHash(string)

## 8. テスト仕様（投票機能）

投票機能に関する主なテスト観点は以下の通り。

- 提案が作成できること
- 作成した提案に対して投票ができること
- ブロックが生成されて作成した投票の状態が変化していること

### 8.1 テストパターン一覧

**コントラクトのテスト**

| テストID | テスト内容 | 期待結果 |
|----------|------------|----------|
| T-01 | 提案作成処理が正常に完了し作成した投票がブロック生成により投票状態が変化すこと | 投票状態（例: Active→Succeeded/Defeated）が正しく遷移する |
| T-02 | 提案作成後に投票ができること | 投票イベントが発生し、投票数が正しく集計される |
| T-03 | 投票結果確定後に賛成多数の場合NFT配布が行われる | 投票参加者にNFTが配布される |
| T-04 | 投票結果確定後に拒否多数の場合NFT配布が行われない | 投票参加者にNFTが配布されない |
| T-05 | 投票期間外の投票は拒否される | 投票期間外は投票できず、エラーとなる |
